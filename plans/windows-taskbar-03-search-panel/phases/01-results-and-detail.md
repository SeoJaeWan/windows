# Phase 1. Search 패널 결과 구조 만들기

> 이 문서는 실행용 상세 계약이다. `plan.md`의 같은 phase 요약을 기술적으로 확장하되, 범위나 결론을 새로 바꾸지 않는다.

- owner_agent: `frontend-developer`
- 목적: `TaskbarSearchPanel`의 primary state를 고정해 Search 결과와 detail pane을 같은 panel owner 안에서 읽히게 만든다.
- boundary:
  - `packages/ui/src/index.ts`
  - `packages/ui/src/components/taskbar/taskbarSearchPanel/**`
  - `packages/ui/src/components/taskbar/internal/**`
  - `plans/windows-taskbar-03-search-panel/reference-captures.md`
- input:
  - 선행 계약:
    - `packages/ui/src/index.ts`와 `packages/ui/src/components/taskbar/**`가 다음 canonical public taskbar contract를 실제 파일 기준으로 제공할 것: `Taskbar`, `TaskbarWindowsButton`, `TaskbarSearch`, `TaskbarIconButton`, `TaskbarClock`, `TaskbarWindowsPanel`, `TaskbarSearchPanel`, `TaskbarHoverPanel`, `TaskbarIconContextMenu`; 그리고 `packages/ui/src/components/taskbar/internal/**`에 taskbar 전용 private primitive가 존재할 것.
  - 기준 캡처:
    - `search-panel-default.png`
    - `search-panel-query-results.png`
    - `search-panel-query-detail.png`
    - `search-panel-query-detail-pinned.png`
    - `search-panel-query-empty.png`
  - 현재 실행 계약:
    - `packages/ui/package.json`의 `test` 스크립트는 `vitest run`
    - TypeScript 검증은 `pnpm --filter @windows/ui exec tsc --noEmit -p tsconfig.json`
- output:
  - 공개 계약:
    - `TaskbarSearchPanel`은 public panel owner이고, panel 내부 검색 input을 새로 만들지 않는다.
    - `TaskbarSearchPanel`은 `default`, `query-results`, `query-detail`, `query-empty` 네 primary state를 정적 UI로 표현할 수 있어야 한다.
    - `default` state는 왼쪽 추천 목록과 오른쪽 featured card 영역을 가진다.
    - `query-results` state는 제목 `최적의 일치`, 세로 결과 목록, 각 결과 행 오른쪽의 disclosure `>` affordance를 가진다.
    - `query-detail` state는 결과 목록을 유지한 채 오른쪽 detail pane을 추가하고, 제목, source label, action list를 가진다.
    - `query-detail` state는 `시작 화면에 고정`과 `시작 화면에서 제거` 라벨 반전을 visual variant로 표현할 수 있어야 한다.
    - `query-empty` state는 query가 표시된 상태의 제목 `최적의 일치`와 빈 결과 영역을 가진다.
  - 내부 기본값:
    - `TaskbarSearchPanel`은 `PanelSurface`, `SectionHeading`, `ListRow`, `TileCard`, `DetailPane`, `ActionList`, `EmptyStateBlock`, `Icon` 계열 private primitive를 조합한다.
    - 결과 행의 disclosure `>`는 generic icon button이 아니라 detail pane을 여는 affordance처럼 보여야 하지만 event API는 가지지 않는다.
    - detail pane의 action glyph는 Fluent action icon 후보를 쓸 수 있다.
  - 허용하지 않는 대안:
    - Search panel 안에 별도 검색 input을 두는 구조
    - 결과 목록과 detail pane을 하나의 generic list 카드로 축소하는 구조
    - `query-detail`의 pin/unpin 라벨 반전을 무시하는 구조
- 선행조건:
  - `plans/windows-taskbar-01-foundation-shell/phases/01-public-contract-and-primitives.md`의 output이 현재 소스 트리에 반영되어 있을 것
  - `packages/ui/src/index.ts`와 `packages/ui/src/components/taskbar/**`가 다음 canonical public taskbar contract를 실제 파일 기준으로 제공할 것: `Taskbar`, `TaskbarWindowsButton`, `TaskbarSearch`, `TaskbarIconButton`, `TaskbarClock`, `TaskbarWindowsPanel`, `TaskbarSearchPanel`, `TaskbarHoverPanel`, `TaskbarIconContextMenu`; 그리고 `packages/ui/src/components/taskbar/internal/**`에 taskbar 전용 private primitive가 존재할 것.
- 제약:
  - 이 단계는 primary state와 detail pane까지만 다룬다.
  - 결과 행 context menu는 다음 phase 범위다.
- failure/validation:
  - Search panel이 panel-search field를 다시 내부에 가지면 실패다.
  - `query-results`와 `query-detail` 차이가 사라져 `>` disclosure 의미가 없어지면 실패다.
  - pin/unpin 라벨 반전이 detail pane에서 표현되지 않으면 실패다.
- 작업:
  - `TaskbarSearchPanel`의 primary state를 `default`, `query-results`, `query-detail`, `query-empty`로 고정한다.
  - 추천 목록과 featured card 영역을 `default` state에 구성한다.
  - 결과 행과 trailing disclosure를 `query-results` state에 구성한다.
  - 제목, source label, action list를 가진 detail pane을 `query-detail` state에 구성한다.
  - pin/unpin 라벨 반전 variant를 `query-detail` state에 반영한다.
  - 빈 결과 영역을 `query-empty` state에 구성한다.
