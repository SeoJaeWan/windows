# Phase 2. 화면별 목록 계산 묶기

> 이 문서는 실행 계약이다. `plan.md`의 같은 phase 요약을 기술적으로 확장하되, 범위와 완료 조건을 바꾸지 않는다.

- owner_agent: `frontend-developer`
- 목적: icon strip, windows surface, search surface를 계산하는 helper 경계를 package-private plain model로 고정한다.
- boundary:
  - `packages/ui/src/utils/taskbar/**`
  - `packages/ui/src/components/taskbar/internal/**`
  - `packages/ui/src/components/taskbar/taskbarStartPanel/**`
  - `packages/ui/src/components/taskbar/taskbarSearchPanel/**`
- input:
  - `plans/windows-taskbar-work-01-data-helper-foundation/phases/01-data-contract.md`에서 고정한 공식 입력과 winner rule
  - 현재 source tree에는 `packages/ui/src/utils/taskbar/**` 경계가 없고, `taskbarStartPanel`과 `taskbarSearchPanel`이 받는 data shape를 누가 계산하는지 닫혀 있지 않다.
  - 현재 `taskbarStartPanel`은 `pinnedItems`, `categories`, `sections`, `resultItems`, `detail` data shape를 요구하고, `taskbarSearchPanel`은 `recommendedItems`, `featuredItems`, `resultItems`, `detail` data shape를 요구한다.
- output:
  - 공개 계약:
    - 후속 일감이 의존할 package-private helper 경계가 source tree에 존재하고, root export와 `@windows/ui/interactive`로 노출되지 않는다.
    - `taskbarStartPanel`은 windows helper가 만든 data part를 소비하고, `taskbarSearchPanel`은 search helper가 만든 data part를 소비한다.
    - `taskbarStartPanel` results mode와 `taskbarSearchPanel` results mode는 같은 `resultItems`/`detail` object contract를 공유한다.
    - helper는 화면에 그릴 data part만 만든다. `placeholder`, selection callback, close callback 같은 render 외 prop 주입은 후속 composite assembly가 맡는다.
  - 내부 기본값:
    - icon helper는 `{ id, label, kind, status?, category, entryId?, reservedLauncher }` shape를 반환한다.
    - icon helper의 `reservedLauncher`는 `icon.category === "windows"`일 때만 `true`다. reserved launcher를 제외한 icon만 `entry.category`와 매핑된다.
    - windows pinned helper는 `taskbarStartPanel` pinned mode data part로 `{ heading, viewAllLabel, pinnedItems }`를 반환한다.
    - `pinnedItems` item shape는 `{ id, label, icon?, description? }`다. 대상은 `entry.windows.pinned === true`인 entry만이며, 정렬은 `entry.windows.order` 오름차순 후 선언 순서 fallback이다. `label`은 `entry.title`, `description`은 `entry.summary ?? entry.meta ?? undefined`다.
    - windows all helper는 `taskbarStartPanel` all mode data part로 `{ categories, sections }`를 반환한다.
    - `categories` item shape는 `{ id, label, active? }`다. `id`는 `entry.category`, `label`은 `entry.category`를 start case로 변환한 문자열, `active`는 첫 번째 category만 `true`다.
    - `sections` item shape는 `{ id, label, items }`다. `id`와 `label`은 같은 category 기준을 재사용하고, `items` item shape는 `{ id, label, icon?, description? }`다.
    - windows all helper는 `entry.windows.visible !== false`인 entry만 포함하고, reserved `windows` launcher category는 category/section 생성 대상에서 제외한다.
    - windows all helper는 visible entry를 먼저 `entry.windows.order` 오름차순 후 선언 순서 fallback으로 정렬한 뒤, 그 순서에서 처음 등장한 `entry.category` 순서대로 `categories`와 `sections`를 만든다. 각 section의 item 순서도 같은 정렬 결과를 따른다.
    - search default helper는 `taskbarSearchPanel` default mode data part로 `{ recommendedItems, featuredItems }`를 반환한다.
    - `recommendedItems` item shape는 `{ id, label, meta?, icon? }`다. 대상은 `entry.search.searchable !== false`이면서 `entry.search.recommended === true`인 entry만이며, 정렬은 `entry.search.rank` 오름차순 후 선언 순서 fallback이다. `label`은 `entry.title`, `meta`는 `entry.meta`, `icon`은 matching icon이다.
    - `featuredItems` item shape는 `{ id, label, description?, thumbnailSrc?, thumbnailAlt? }`다. 대상은 `entry.search.searchable !== false`이면서 `entry.search.featured === true`인 entry만이며, 정렬은 `entry.search.rank` 오름차순 후 선언 순서 fallback이다. `label`은 `entry.title`, `description`은 `entry.summary ?? entry.meta ?? undefined`다.
    - search results helper는 `taskbarSearchPanel` results mode data part로 `{ query, resultItems, detail }`를 반환한다.
    - `query`는 `search.value ?? ""`이고, `resultItems`와 `detail`은 public `search.resultItems`, `search.detail`을 그대로 전달한다.
    - search results helper는 `entries`에서 결과를 검색하거나 rank를 다시 계산하지 않고, `resultItems[].active`와 `detail`의 관계를 내부에서 보정하지 않는다.
  - 허용하지 않는 상태:
    - helper가 JSX, DOM prop bundle, preview wrapper 책임까지 같이 반환하는 구조
    - windows all grouping이 `entry.category` 외의 두 번째 taxonomy를 요구하는 구조
    - search results mode가 `entries`만으로 결과 목록이나 detail을 다시 계산하는 구조
    - route fixture나 story data가 helper contract를 대신 설명하는 구조
    - helper가 root export나 public import path로 노출되는 구조
- 선행조건: `plans/windows-taskbar-work-01-data-helper-foundation/phases/01-data-contract.md` 완료
- 제약:
  - hover/context projection은 이 일감의 필수 산출물이 아니다. 해당 surface가 같은 registry를 쓰는 여부는 후속 조합 일감에서 결정한다.
  - 이 phase는 package-only helper 경계만 닫는다. `apps/web`, Storybook, visuals는 수정 대상이 아니다.
- failure/validation: helper output field set이 recipient panel shape와 어긋나거나, results payload를 helper 내부에서 다시 해석하게 되면 실패다.
- 작업:
  - `utils/taskbar` 또는 동등한 package-private 경계를 추가해 icon strip, windows, search 계산 책임을 package 안으로 모은다.
  - `taskbarStartPanel`과 `taskbarSearchPanel`이 요구하는 data part field set을 helper output 계약으로 source tree 기준에 맞춰 고정한다.
  - `entry.category` 기반 windows all grouping, `search.recommended`/`search.featured` 기반 default projection, external controlled results payload passthrough를 helper 책임으로 명시한다.
  - helper가 public export, sandbox fixture, story data로 계약을 우회하지 않도록 경계를 닫는다.
